# LeetCode 1079 - Letter Tile Possibilities

使用 C# 與 .NET 10 實作 LeetCode 1079 活字印刷。
這個專案的核心目標，是計算一組字母磁磚可以組成多少個非空字母序列，並示範兩種等價的回溯寫法：

- 用 Dictionary<char, int> 管理字元次數。
- 用只保留剩餘次數的陣列壓縮搜尋狀態。

> [!TIP]
> 這題的重點不是把所有排列真的組成字串，而是追蹤每個字元剩餘幾次可用。這樣每一層遞迴只要決定「下一個要拿哪個字元」，就能自然避開重複分支。

## 題目說明

給定一個字串 tiles，其中每個字元代表一塊印有字母的磁磚。每塊磁磚最多只能使用一次，請回傳可以組成的所有非空字母序列數量。

題目連結：

- LeetCode: [Letter Tile Possibilities](https://leetcode.com/problems/letter-tile-possibilities/description/)
- 力扣中文站: [活字印刷](https://leetcode.cn/problems/letter-tile-possibilities/description/)

以 tiles = AAB 為例，可組成的非空序列包含：

- 長度 1: A、B
- 長度 2: AA、AB、BA
- 長度 3: AAB、ABA、BAA

總數為 8。

## 解題概念與出發點

如果直接把所有字元做全排列，會立刻遇到兩個問題：

1. 有重複字元時，會產生大量重複排列。
2. 題目要求的是所有非空序列，不只固定長度的排列。

因此這題更適合把問題改寫成：

「目前還有哪些字元可以選？每種字元還剩幾次？」

一旦用這個角度思考，回溯流程就很清楚：

1. 先統計每個字元的出現次數。
2. 每一層遞迴嘗試挑選一個仍有剩餘次數的字元。
3. 挑選後扣掉次數，繼續往下搜尋更長的序列。
4. 搜尋完畢後把次數加回來，恢復狀態。

這樣做有兩個直接好處：

- 不需要真的建立所有字串，只要累加可行狀態數量。
- 同一個字母只會以「剩餘次數」的形式管理，不會因為相同字元位置不同而重複計算。

除了用 Dictionary<char, int> 直接管理字元與次數，這題還有一個很實用的觀察：

- 題目中的字元都是大寫英文字母。
- 真正參與回溯的核心資訊，其實只有「每種字元還剩幾次」。

因此只要先把每種不同字元的次數抽出來，就可以把狀態壓縮成一個整數陣列。例如 tiles = AAB 時：

- 用字典表示是 A: 2、B: 1
- 用次數陣列表示就是 [2, 1]

這種寫法不再關心陣列位置對應哪個實際字母，而是只關心「某一類字元還有沒有剩」，因此同樣可以正確去重。對這題來說，只要求數量、不需要列出所有序列時，這是很自然的另一種實作方式。

> [!NOTE]
> 在這份實作中，遞迴函式會把「目前路徑本身」也算成 1 個結果，所以最外層最後要扣掉空字串，答案才會符合題目要求的非空序列數量。

## 回溯法核心觀念

在進入這題的兩種實作之前，先把「回溯法」本身拆開來看，會更容易理解為什麼這題自然適合用 DFS 加上回復狀態來解。

很多人第一次接觸回溯法時，會把它想成「一直試錯，錯了就退回去」。這個描述方向沒有錯，但還不夠精準。更完整的理解應該是：

- 回溯法是一種有系統地枚舉所有可能解的方法。
- 它不是亂試，而是沿著搜尋樹一層一層展開選擇。
- 每往下一層，代表多做了一個決策。
- 當某條路徑已經不能繼續，或已經得到一個完整答案，就停止往下展開。
- 返回上一層時，要把剛才做過的修改恢復，讓其他分支能在乾淨狀態下繼續搜尋。

如果把整個過程濃縮成一句話，就是：

「選擇一個可能動作，暫時套用它，深入探索結果，再把動作撤銷，換試下一個動作。」

### 1. 回溯法的核心目標是什麼

回溯法最核心的目標，不是單純把遞迴寫出來，而是用一套可控的方式處理這類問題：

- 所有可能解很多，無法手動列舉。
- 每一步都有多個選擇。
- 前面做的選擇，會影響後面還能做哪些選擇。
- 題目要你找到所有合法答案、統計合法答案數量，或找出某個符合條件的答案。

因此回溯法真正想解決的是：

1. 如何把大問題拆成一連串小決策。
2. 如何有系統地走遍每一條合法路徑。
3. 如何在走完某條路之後，不污染其他分支。

對這題來說，目標就是把「組出一個非空字母序列」拆成多次決策：

- 第一步選哪個字元。
- 第二步在剩下的字元裡再選哪個。
- 第三步繼續選。
- 一直選到沒有字元可用為止。

換句話說，回溯法不是直接思考「最後有哪些字串」，而是思考「從現在這個狀態開始，下一步可以怎麼走」。

### 2. 用搜尋樹來看回溯法

理解回溯法最好的方式，就是把它想成一棵搜尋樹。

在搜尋樹裡：

- 每個節點代表一個「目前狀態」。
- 每條邊代表一個「做出的選擇」。
- 從根節點走到某個節點的整條路徑，代表一個「目前已形成的部分答案」。

以這題為例：

- 根節點是空字串，表示還沒選任何字元。
- 從根節點走到 A，表示第一步選了 A。
- 從 A 再走到 AB，表示第二步又選了 B。
- 從 AB 再走到 ABA，表示第三步又選了 A。

所以回溯法的本質，就是在這棵樹上做深度優先搜尋：

- 先一路往下走，盡量延伸當前路徑。
- 走到底之後退回上一層。
- 改走另一條分支。
- 直到整棵樹走完。

這個視角很重要，因為它能幫你區分兩件事：

- 我現在手上保存的是什麼狀態。
- 我下一步能擴展哪些分支。

只要這兩件事定義清楚，回溯法通常就能自然寫出來。

### 3. 回溯法的四個固定元素

大多數回溯題，本質上都要先想清楚四件事。

#### 狀態

目前遞迴走到哪裡，以及後面還剩哪些資源可以使用。

在這題裡，狀態不是「目前用了原字串第幾個位置」，而是：

- 目前已經形成了多長的前綴。
- 每種字元還剩幾次可以用。

#### 選擇

從目前狀態出發，下一步有哪些合法動作。

在這題裡，選擇就是：

- 任選一個剩餘次數大於 0 的字元。

#### 約束

哪些選擇不能做，或做了之後什麼情況要停止。

在這題裡，最直接的約束是：

- 某個字元次數若已經是 0，就不能再選。

#### 答案累積方式

要在什麼時機把目前路徑算進答案。

這題的特別之處在於：

- 只要目前路徑不是空的，它就是一個合法序列。
- 所以不必等到長度固定，或所有字元都用完才記錄答案。

這也是為什麼這題的回溯邏輯，和常見的「找固定長度排列」不完全一樣。

### 4. 為什麼一定要做「選擇、遞迴、回復」

回溯法最標誌性的部分，就是這三個動作永遠一起出現：

1. 做出一個選擇。
2. 帶著這個選擇進入下一層。
3. 回來時撤銷這個選擇。

標準骨架可以寫成：

```text
Backtrack(狀態):
   如果目前狀態已符合記錄條件:
      累加答案

   對每個可做的選擇:
      做選擇
      Backtrack(新狀態)
      撤銷選擇
```

這裡最容易被低估的，是最後一步「撤銷選擇」。

因為回溯法通常會共用同一份資料結構，例如：

- 陣列
- 字典
- 路徑串列
- 已使用標記

如果你在某個分支把狀態改掉，回來之後卻不恢復，那麼下一個兄弟分支就不是從原本的狀態出發，而是從一個被污染的狀態出發，答案自然會錯。

對這題來說，回復狀態就是：

- 選了某個字元後，把它的剩餘次數減一。
- 該分支搜尋完成後，再把次數加回來。

少了這一步，其他分支看到的就不是正確的剩餘字元數量。

### 5. 這題如何完整映射成回溯模型

把題目轉成回溯模型後，整體會非常清楚。

#### 狀態是什麼

- 目前路徑已經形成了什麼前綴。
- 每種字元還剩幾次。

#### 每一層在做什麼

- 嘗試把某個仍可使用的字元接到目前前綴後面。

#### 為什麼這題不需要真的組出所有字串

因為題目只要求數量，不要求輸出所有序列內容。

所以實際上，回溯時真正有價值的不是整個字串本身，而是：

- 現在這個前綴是否存在。
- 加入下一個字元後，剩餘資源如何改變。

只要能正確描述「還剩哪些字元可用」，就能計算答案。這也是為什麼這題可以只靠次數表，甚至只靠次數陣列來完成搜尋。

#### 什麼時候把答案加一

這題每往下成功選到一個字元，就代表形成了一個新的非空前綴，因此可以記成一個答案。

例如：

- 選到 A，得到 A，這要算一次。
- 再選到 B，得到 AB，這也要算一次。
- 再選到 A，得到 ABA，這也要算一次。

因此這題的重點不是「走到葉節點才算答案」，而是「路徑上每一個非空節點都算答案」。

### 6. 這題最關鍵的去重原理

這題之所以適合用回溯，不只是因為能遞迴列舉，更重要的是它能自然處理重複字元。

如果你直接拿原字串的每個位置做排列，例如把兩個 A 視為不同位置：

- 第一個 A + B
- 第二個 A + B

你會得到表面上不同、實際上完全相同的序列，最後還要額外去重。這會讓搜尋空間膨脹很多。

真正正確的做法，是把問題提升到「字元種類」層級：

- 不問你選的是第幾個 A。
- 只問 A 還剩幾個可以選。

一旦狀態改成這樣，重複問題就自然消失：

- A 還剩 2 個，是一種狀態。
- A 還剩 1 個，是另一種狀態。

同一種剩餘次數狀態，不會因為原字串裡相同字元來自不同索引而分裂成多條重複分支。

這也是本題最核心的建模技巧。

### 7. 為什麼用字典與用次數陣列，本質上是同一件事

README 裡展示了兩種寫法，但它們的回溯原理完全一致。

第一種寫法保留：

- 哪個字元
- 這個字元剩幾次

第二種寫法只保留：

- 某一類字元剩幾次

差別只在狀態表示法，不在演算法本質。

也就是說，兩者都在做同一件事：

1. 找出目前還能選的分支。
2. 消耗一個資源。
3. 深入下一層。
4. 返回後恢復資源。

第二種寫法之所以成立，是因為這題只需要「數量」，不需要真的知道當前選的是字母 A 還是 B。對計數而言，剩餘次數結構才是核心資訊。

### 8. 這題的終止條件與剪枝觀念

回溯題常常會談剪枝，但這題的剪枝其實很自然，也很輕量。

#### 終止條件

當所有可選字元都已經耗盡，或 remainingLength 變成 0，就不能再往下擴展。

#### 隱含剪枝

只要某個字元剩餘次數是 0，就直接跳過，不開新的分支。

這雖然看起來簡單，但本質上就是剪枝，因為你避免進入不可能成功的路徑。

這題不需要非常複雜的額外剪枝技巧，原因是：

- 狀態本身已經壓縮得很乾淨。
- 重複分支已經透過「只看剩餘次數」被消掉。

換句話說，好的狀態設計，本身就是最有效的剪枝。

### 9. 學回溯法時最該抓住的幾個關鍵點

如果要把這一章濃縮成真正有用的判斷原則，最值得記住的是下面幾點：

1. 回溯法的本質是搜尋樹，不是單純把遞迴硬寫出來。
2. 先定義狀態，再定義選擇，最後才寫程式。
3. 狀態設計得好，重複分支和剪枝問題通常會一起變簡單。
4. 共享狀態就一定要回復，不然兄弟分支會互相污染。
5. 這題的答案不是只出現在最深層，而是每個非空前綴都算一個答案。
6. 去重的根本方法，不是最後再用集合硬去掉，而是一開始就避免把相同字元的不同位置當成不同分支。

### 10. 用一句話總結這題的回溯核心

對 LeetCode 1079 來說，回溯法的核心可以總結成：

「把問題改寫成剩餘字元次數的搜尋，對每一個仍可使用的字元展開分支，走完後恢復狀態，並把每個新形成的非空前綴都計入答案。」

只要抓住這個觀念，後面的字典版與次數陣列版，其實都只是同一套搜尋邏輯的不同狀態表示。

## 解法詳解

### 1. 建立字元次數表

先掃描整個 tiles 字串，把每個字元出現幾次記錄到 Dictionary<char, int> 中。

範例：

- 輸入 AAB
- 次數表會變成 A: 2、B: 1

這一步的目的，是把原始輸入壓縮成「選擇空間」。

### 2. 只在不重複字元之間展開分支

有了次數表之後，遞迴每一層只需要遍歷不重複字元集合，例如 A 和 B。

對每個字元：

- 如果剩餘次數是 0，代表不能再選，直接跳過。
- 如果剩餘次數大於 0，代表可以拿來放到目前序列後面。

### 3. 選擇、遞迴、回復

這是典型回溯三步驟：

1. 選擇目前字元，將次數減一。
2. 遞迴搜尋下一層。
3. 返回後把次數加回去，讓其他分支可以共用同一份狀態。

這裡的關鍵在於「回復現場」，因為每一層都共用同一份次數表。如果不回復，後面的搜尋結果就會被前面的分支污染。

### 4. 為什麼每層先從 1 開始累加

DFS 內部一開始先把答案設成 1，代表：

- 即使這一層不再往下擴展，當前前綴本身也是一種有效狀態。

例如前綴已經是 AB，那麼 AB 本身就應該計入答案，之後再看是否還能繼續接其他字元。

這個寫法可以把「不同長度序列都要計數」的需求整合進同一套遞迴中，不需要另外針對長度 1、2、3 分開處理。

### 5. 遞迴終止條件

當 remainingLength 變成 0 時，代表這條路徑已經不能再放更多字元，直接回傳 1。

這個 1 表示：

- 目前這個狀態是一個完成展開的可行結果。

## 解法二詳解

第二種寫法保留同樣的回溯核心，但把「字元次數表」進一步壓縮成「次數陣列」。

### 1. 先統計每種字元出現幾次

雖然最後回溯會用陣列處理，但一開始仍然可以先掃描 tiles，統計每個字元的出現次數。

例如 tiles = AAB 時：

- A 出現 2 次
- B 出現 1 次

### 2. 只保留次數，不保留字元本身

因為題目只要求「有多少種非空序列」，不要求把每個序列真的列出來，所以回溯真正需要知道的是：

- 目前有幾類字元可以選
- 每一類字元還剩幾次

因此可以把 A: 2、B: 1 轉成：

- counts = [2, 1]

假設第 0 格代表某一類字元、第 1 格代表另一類字元即可，不需要在遞迴過程中反覆保留字元內容。

### 3. 每次選一個剩餘次數大於 0 的位置

回溯時，依序檢查 counts 中每個位置：

- 若 counts[i] = 0，代表這一類字元已用完，不能再選。
- 若 counts[i] > 0，代表可以再放入一個字元。

一旦選了某個位置，就做三件事：

1. 將 counts[i] 減一。
2. 因為剛加入了一個字元，所以答案先加 1。
3. 繼續往下回溯，計算後面能接出的更長序列。

### 4. 回溯後要把次數加回來

和一般回溯一樣，遞迴結束後必須把 counts[i] 加回去，恢復原本狀態。

如果不恢復，下一個分支就會看到被前一條路徑改壞的剩餘次數，導致答案錯誤。

### 5. 為什麼這樣不會重複計算

重點在於整個搜尋過程不是從「原字串中第幾個位置的字元」出發，而是從「某一類字元還剩幾個」出發。

以兩個 A 為例：

- 我們不會分別把它們視為第一個 A、第二個 A。
- 我們只會視為「A 還剩 2 個」或「A 還剩 1 個」。

因此相同字元不會因為來源位置不同而形成重複分支。

## 範例演示

以下用 tiles = AAB 說明實際搜尋流程。

初始狀態：

- 次數表：A: 2、B: 1
- 可選字元：A、B
- 呼叫 DFS(3, count, uniqueTiles)

### 搜尋樹概念

```text
空字串
├─ A
│  ├─ AA
│  │  └─ AAB
│  └─ AB
│     └─ ABA
└─ B
   └─ BA
      └─ BAA
```

搭配長度 1 的節點一起看：

- A
- B
- AA
- AB
- BA
- AAB
- ABA
- BAA

總共有 8 個非空序列。

### 逐步說明

1. 第一層可以先選 A，也可以先選 B。
2. 若先選 A，次數表變成 A: 1、B: 1。
3. 第二層仍可選 A 或 B。
4. 若第二層再選 A，就得到前綴 AA，接著第三層只能選 B，得到 AAB。
5. 若第二層改選 B，就得到前綴 AB，接著第三層只能選 A，得到 ABA。
6. 若第一層先選 B，次數表變成 A: 2、B: 0。
7. 第二層只能選 A，得到 BA；第三層再選 A，得到 BAA。
8. 所有分支完成後，再扣掉一開始代表空字串的那個結果，最終答案是 8。

### 解法二的次數陣列流程

如果改用次數陣列表示，tiles = AAB 會先被壓縮成：

- counts = [2, 1]

這裡可以把它理解成：

- 第 0 類字元剩 2 次
- 第 1 類字元剩 1 次

接著回溯會依序走過以下狀態。

1. 先選第 0 類字元，形成「A」，counts 變成 [1, 1]。
2. 再選第 0 類字元，形成「AA」，counts 變成 [0, 1]。
3. 再選第 1 類字元，形成「AAB」，counts 變成 [0, 0]。
4. 回上一層後改選第 1 類字元，形成「AB」，counts 變成 [1, 0]。
5. 再選第 0 類字元，形成「ABA」，counts 變成 [0, 0]。
6. 回到起點後，先選第 1 類字元，形成「B」，counts 變成 [2, 0]。
7. 接著選第 0 類字元，形成「BA」，counts 變成 [1, 0]。
8. 最後再選第 0 類字元，形成「BAA」，counts 變成 [0, 0]。

因此會依序計到這 8 個非空序列：

- A
- AA
- AAB
- AB
- ABA
- B
- BA
- BAA

可以看到，雖然程式內部只操作 [2, 1] 這種剩餘次數狀態，實際上對應到的搜尋結果和解法一完全一致。

## 複雜度分析

- 時間複雜度：與實際展開的可行序列數量成正比；在最壞情況下可視為接近排列總數等級。
- 空間複雜度：O(n + k)，其中 n 是遞迴深度，k 是不重複字元數量。
