# LeetCode 997 - Find the Town Judge

使用 .NET Console 專案整理 LeetCode 997「找到小鎮的法官」的兩種線性解法：入度／出度統計，以及單一信任分數統計。專案提供可直接執行的固定案例，方便從有向圖概念開始理解，並核對兩種方法是否得到相同答案。

## 題目連結

- [LeetCode 997. Find the Town Judge](https://leetcode.com/problems/find-the-town-judge/description/)
- [LeetCode CN 997. 找到小鎮的法官](https://leetcode.cn/problems/find-the-town-judge/description/)

## 題目說明

小鎮中有 `n` 個人，編號為 `1` 到 `n`。如果小鎮法官存在，他必須同時符合三個條件：

1. 法官不信任任何人。
2. 除了法官本人以外，每個人都信任法官。
3. 只有一個人同時符合前面兩個條件。

輸入 `trust[i] = [a, b]` 表示人物 `a` 信任人物 `b`。如果能唯一找出法官，就回傳法官的編號；否則回傳 `-1`。

## 限制條件

- `1 <= n <= 1000`
- `0 <= trust.length <= 10^4`
- `trust[i].length == 2`
- `1 <= a, b <= n`
- `a != b`，沒有人信任自己
- 所有 `[a, b]` 信任關係都不重複

## 解題概念與出發點

### 先把信任關係看成有向圖

把每個人看成一個節點，`[a, b]` 則是一條從 `a` 指向 `b` 的有向邊：

```text
a 信任 b
a --------> b
```

以 `n = 3, trust = [[1, 3], [2, 3]]` 為例：

```text
1 -----> 3 <----- 2
```

人物 `3` 被另外兩個人信任，而且沒有任何箭頭從 `3` 指向別人，因此 `3` 是法官。

### 法官條件如何轉成可計算的數字

- 一個人被信任一次，他的「入度」增加 1。
- 一個人信任別人一次，他的「出度」增加 1。
- 法官必須被其餘 `n - 1` 人信任，所以入度為 `n - 1`。
- 法官不能信任任何人，所以出度為 `0`。

這就是第一種解法的直接判斷式：

```text
inDegree[i] == n - 1 && outDegree[i] == 0
```

第二種解法再把這兩個條件壓縮成一個分數：被信任加 1 分，信任別人減 1 分。法官最後一定得到 `n - 1` 分。

## 解法比較

令 `m = trust.length`：

| 解法 | 核心資料 | 判斷條件 | 時間複雜度 | 空間複雜度 |
| --- | --- | --- | --- | --- |
| `FindJudge` | 入度陣列、出度陣列 | 入度為 `n - 1` 且出度為 `0` | `O(n + m)` | `O(n)` |
| `FindJudgeByTrustScore` | 單一信任分數陣列 | 分數為 `n - 1` | `O(n + m)` | `O(n)` |

兩種方法都只需要：

1. 掃描全部 `m` 筆信任關係。
2. 掃描 `1` 到 `n` 的人物編號尋找答案。

差別只在於第一種方法分開保存兩個條件，第二種方法把兩個條件合併成一個分數。

## 解法一：入度與出度陣列 `FindJudge`

### 設計說明

建立兩個長度為 `n + 1` 的陣列，讓陣列索引直接對應人物編號：

- `inDegrees[i]`：有多少人信任人物 `i`。
- `outDegrees[i]`：人物 `i` 信任多少人。

讀到 `[a, b]` 時：

```text
outDegrees[a]++
inDegrees[b]++
```

全部關係統計完成後，從人物 `1` 掃描到人物 `n`，尋找同時符合以下條件的人：

```text
inDegrees[i] == n - 1
outDegrees[i] == 0
```

如果找不到，就回傳 `-1`。

### 為什麼有效

入度 `n - 1` 證明除了自己以外，每個人都信任這位候選人；出度 `0` 則證明候選人沒有信任任何人。兩個條件一起檢查，正好完整對應題目對法官的定義。

題目保證信任關係不重複，因此不會因為同一個人重複提供相同信任而把入度錯誤地累加到 `n - 1`。

### 範例演示流程

使用 `n = 3, trust = [[1, 3], [2, 3]]`：

初始時所有入度與出度都是 `0`。

| 處理的關係 | 變化 | 人物 1 `(入, 出)` | 人物 2 `(入, 出)` | 人物 3 `(入, 出)` |
| --- | --- | --- | --- | --- |
| 尚未處理 | - | `(0, 0)` | `(0, 0)` | `(0, 0)` |
| `[1, 3]` | 1 的出度 +1，3 的入度 +1 | `(0, 1)` | `(0, 0)` | `(1, 0)` |
| `[2, 3]` | 2 的出度 +1，3 的入度 +1 | `(0, 1)` | `(0, 1)` | `(2, 0)` |

因為 `n - 1 = 2`，人物 `3` 的入度為 `2`、出度為 `0`，所以回傳 `3`。

### 優缺點

- 優點：資料含義直接，能清楚看到每個人的入度與出度，最適合第一次學習圖論的人。
- 優點：判斷式與題目兩個法官條件一一對應，容易驗證正確性。
- 代價：需要兩個長度為 `n + 1` 的陣列；雖然仍是 `O(n)` 空間，但可以再壓縮。

## 解法二：單一信任分數 `FindJudgeByTrustScore`

### 設計說明

觀察法官的兩個條件：

- 被別人信任是正面條件，每次加 `1` 分。
- 信任別人會違反法官條件，每次減 `1` 分。

因此讀到 `[a, b]` 時：

```text
trustScores[a]--
trustScores[b]++
```

法官被另外 `n - 1` 人信任，又完全沒有信任別人，所以最後分數一定是 `n - 1`。

### 為什麼只檢查分數就足夠

某位不是法官的人即使被很多人信任，只要他至少信任一個人，就會被扣分。由於一個人最多只能被其餘 `n - 1` 人信任，他在有出度的情況下不可能仍得到 `n - 1` 分。

因此，在題目保證關係唯一且不允許自己信任自己的前提下，`trustScores[i] == n - 1` 已同時代表：

- 人物 `i` 被其餘所有人信任。
- 人物 `i` 沒有信任任何人。

### 範例演示流程

使用同一組 `n = 3, trust = [[1, 3], [2, 3]]`：

| 處理的關係 | 人物 1 分數 | 人物 2 分數 | 人物 3 分數 |
| --- | ---: | ---: | ---: |
| 尚未處理 | `0` | `0` | `0` |
| `[1, 3]` | `-1` | `0` | `1` |
| `[2, 3]` | `-1` | `-1` | `2` |

`n - 1 = 2`，人物 `3` 的分數正好是 `2`，所以回傳 `3`。

再看容易誤判的案例 `n = 3, trust = [[1, 3], [2, 3], [3, 1]]`：

| 處理完成後 | 人物 1 分數 | 人物 2 分數 | 人物 3 分數 |
| --- | ---: | ---: | ---: |
| 最終分數 | `0` | `-1` | `1` |

人物 `3` 雖然被人物 `1`、`2` 信任，但他也信任人物 `1`，因此被扣 1 分，最後只有 `1` 分，不符合 `n - 1 = 2`。這組資料沒有法官，所以回傳 `-1`。

### 優缺點

- 優點：只用一個陣列，同時表達「被信任」與「信任別人」兩個條件。
- 優點：程式碼短，仍維持 `O(n + m)` 時間複雜度。
- 代價：分數的意義不像入度／出度那麼直觀，需要先理解加分與扣分為什麼能合併條件。

## 特殊案例：為什麼 `n = 1` 的答案是 `1`

當小鎮只有人物 `1` 時，`trust` 是空陣列：

- 人物 `1` 不信任任何人。
- 「除了法官本人以外，每個人都信任法官」沒有其他人需要檢查。
- `n - 1 = 0`，人物 `1` 的入度與信任分數也都是 `0`。

因此兩種解法都會回傳 `1`。
