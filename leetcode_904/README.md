# 904. Fruit Into Baskets（水果成籃）

> LeetCode 第 904 題 — 滑動窗口經典題型

- [English Description (LeetCode)](https://leetcode.com/problems/fruit-into-baskets/description/)
- [中文題目 (力扣)](https://leetcode.cn/problems/fruit-into-baskets/description/)

## 題目說明

你正在探訪一家農場，農場從左到右種植了一排果樹。這些樹用整數陣列 `fruits` 表示，其中 `fruits[i]` 是第 `i` 棵樹所產的**水果種類**。

你想要儘可能多地收集水果，但必須遵守以下規則：

1. **只有兩個籃子**，每個籃子只能裝**單一種類**的水果，數量不限。
2. 從任意一棵樹開始，你必須**逐棵向右**摘取恰好一個水果。
3. 一旦遇到無法放入任何籃子的水果（第三種水果），就必須**停止**。

**目標**：回傳你最多能摘取的水果數量。

### 約束條件

- `1 <= fruits.length <= 10^5`
- `0 <= fruits[i] < fruits.length`

### 範例

| 輸入 | 輸出 | 說明 |
|------|------|------|
| `[1,2,1]` | `3` | 只有兩種水果，全部摘取 |
| `[0,1,2,2]` | `3` | 從第 2 棵樹開始摘取 `[1,2,2]` |
| `[1,2,3,2,2]` | `4` | 從第 2 棵樹開始摘取 `[2,3,2,2]` |

## 解題概念與出發點

### 問題轉化

把題目抽象化後，本質上就是：

> **在整數陣列中，找到一個最長的連續子陣列，使得子陣列內最多只包含兩種不同的數字。**

這是一個經典的**滑動窗口（Sliding Window）**問題。

### 為什麼選擇滑動窗口？

- 我們要找的是**連續子陣列**的最大長度。
- 窗口的合法條件（水果種類 ≤ 2）可以透過**雜湊表**快速維護。
- 右邊界不斷擴展、左邊界按需收縮 → 典型的**雙指標**模式。
- 時間複雜度可以做到 **O(n)**，遠優於暴力法的 O(n²)。

## 解法詳解：滑動窗口 + 雜湊表

### 演算法步驟

1. 初始化雙指標 `left = 0`、`right = 0`，以及雜湊表 `cnt`（記錄窗口內每種水果的出現次數）。
2. 每次迴圈，將 `right` 右移一格，把 `fruits[right]` 加入 `cnt`。
3. 若 `cnt` 中的鍵數量超過 2（代表窗口內超過兩種水果）：
   - 將 `fruits[left]` 在 `cnt` 中的計數減 1。
   - 若減至 0，從 `cnt` 中移除該鍵。
   - `left` 右移，持續收縮直到窗口合法。
4. 每次擴展後，更新答案 `res = max(res, right - left + 1)`。
5. 遍歷完成後，`res` 即為答案。

### 複雜度分析

| 項目 | 複雜度 |
|------|--------|
| 時間複雜度 | **O(n)** — `left` 和 `right` 各最多移動 n 次 |
| 空間複雜度 | **O(1)** — 雜湊表中最多存放 3 個鍵值對 |

## 演示流程

以 `fruits = [1, 2, 3, 2, 2]` 為例，逐步演示滑動窗口的運作：

```
初始狀態: left=0, res=0, cnt={}

Step 1: right=0, fruits[0]=1
  cnt = {1:1}           種類數=1 ≤ 2 ✓
  窗口 [1]              長度=1, res=1

Step 2: right=1, fruits[1]=2
  cnt = {1:1, 2:1}      種類數=2 ≤ 2 ✓
  窗口 [1, 2]           長度=2, res=2

Step 3: right=2, fruits[2]=3
  cnt = {1:1, 2:1, 3:1} 種類數=3 > 2 ✗ → 收縮左邊界
    left=0: cnt[1]-- → cnt={2:1, 3:1}  (1 的計數歸零，移除)
    left=1
  cnt = {2:1, 3:1}      種類數=2 ≤ 2 ✓
  窗口 [2, 3]           長度=2, res=2

Step 4: right=3, fruits[3]=2
  cnt = {2:2, 3:1}      種類數=2 ≤ 2 ✓
  窗口 [2, 3, 2]        長度=3, res=3

Step 5: right=4, fruits[4]=2
  cnt = {2:3, 3:1}      種類數=2 ≤ 2 ✓
  窗口 [2, 3, 2, 2]     長度=4, res=4

最終答案: res = 4
```

> [!TIP]
> 核心觀察：`left` 和 `right` 都只會**向右移動**，永遠不會回退。因此整個演算法只需遍歷陣列一次，時間複雜度為 O(n)。

## 補充說明

### 雜湊表計數的簡潔寫法

在解法中，加入新水果至雜湊表時採用了「先判斷是否存在、再遞增」的防禦性寫法：

```csharp
// 原始寫法：明確初始化再遞增
if (!cnt.ContainsKey(fruits[right]))
{
    cnt[fruits[right]] = 0;
}
cnt[fruits[right]]++;
```

在 C# 中，以下三種寫法與上述行為完全等效，可依個人風格選用：

```csharp
// 寫法一：TryGetValue 取出現有值，不存在則預設為 0
cnt.TryGetValue(fruits[right], out int val);
cnt[fruits[right]] = val + 1;

// 寫法二：GetValueOrDefault 取值（更簡潔，一行完成）
cnt[fruits[right]] = cnt.GetValueOrDefault(fruits[right]) + 1;

// 寫法三：TryAdd 嘗試插入初始值，再遞增（意圖最明確）
cnt.TryAdd(fruits[right], 0);
++cnt[fruits[right]];
```

四種寫法的行為完全一致，差異在可讀性、查找次數與 C# 慣用風格：

| 寫法 | 雜湊表查找次數 | 特點 |
|------|:---:|------|
| `ContainsKey` + `++` | 3 次 | 原始防禦性寫法，最易理解 |
| `TryGetValue` + 賦值 | 2 次 | 明確取出舊值再寫入 |
| `GetValueOrDefault` + 賦值 | 2 次 | 現代 C# 推薦，最簡潔 |
| `TryAdd` + `++` | 2 次 | 「初始化」與「遞增」意圖分離，語意清晰 |

> [!NOTE]
> `TryAdd(key, value)` 若鍵已存在則**不覆蓋**直接回傳 `false`，此處忽略回傳值，行為等同「不存在時才初始化為 0」。`GetValueOrDefault` 則是現代 C# 最推薦的一行簡潔做法。
