## 題目說明

給定整數陣列 `heights`，其中 `heights[i]` 表示第 `i` 棟建築的高度。你從索引 `0` 出發，每次只能前往下一棟建築：

- 若下一棟建築不高於目前建築，可以直接前進，不消耗資源。
- 若下一棟比較高，設高度差為 `difference = heights[i + 1] - heights[i]`，必須使用 `difference` 塊磚塊，或使用一把梯子。

目標是在最佳配置磚塊與梯子的前提下，回傳能到達的最遠建築索引。

題目連結：[1642. Furthest Building You Can Reach](https://leetcode.com/problems/furthest-building-you-can-reach/description/)

### 官方範例

| 範例 | `heights` | `bricks` | `ladders` | 輸出 |
| --- | --- | ---: | ---: | ---: |
| 1 | `[4,2,7,6,9,14,12]` | `5` | `1` | `4` |
| 2 | `[4,12,2,7,3,18,20,3,19]` | `10` | `2` | `7` |
| 3 | `[14,3,19,3]` | `17` | `0` | `3` |

## 限制條件

- `1 <= heights.length <= 100000`
- `1 <= heights[i] <= 1000000`
- `0 <= bricks <= 1000000000`
- `0 <= ladders <= heights.length`

輸入陣列最多有十萬個元素，不能枚舉每次上升要選磚塊或梯子的所有組合。解法需要在由左到右掃描時，快速決定哪些高度差值得使用梯子。

## 解題概念與出發點

### 只需要處理正高度差

從 `heights[i - 1]` 前往 `heights[i]` 時：

```text
difference = heights[i] - heights[i - 1]
```

- `difference <= 0`：下一棟沒有更高，可以免費前進。
- `difference > 0`：必須使用等量磚塊或一把梯子。

因此演算法只需要記錄正高度差，不必把下降或等高的移動放進 PriorityQueue。

### 為什麼梯子應該留給較大的高度差

梯子不論跨越高度差 `1` 或 `1000000` 都只消耗一把；磚塊成本卻等於高度差。假設有兩次攀升 `small < large`，但把梯子用在 `small`、磚塊用在 `large`，交換兩種資源後：

```text
原配置的磚塊成本 = large
交換後的磚塊成本 = small
節省的磚塊       = large - small
```

交換後不會增加梯子數，卻能留下更多磚塊。因此在任何已走過的區間中，總能把梯子調整到最大的幾次攀升；其餘較小攀升才使用磚塊。兩個解法都利用這個貪心性質，只是做決策的方向不同：

- 解法一先假設全部使用磚塊，必要時再用梯子取代最大的磚塊支出。
- 解法二先把梯子名額留給高度差，名額超出時把最小高度差改成磚塊。

## 解法一：最大堆，先用磚塊再替換

對應方法：`FurthestBuilding(int[] heights, int bricks, int ladders)`

### 設計說明

這個解法採取「先付磚塊、資源不足時回頭調整」的策略：

1. 從左到右計算相鄰建築的高度差。
2. 遇到非正高度差時直接前進。
3. 遇到正高度差時，先將它視為磚塊支出：扣除 `difference` 塊磚塊，並把高度差加入最大堆。
4. 如果磚塊變成負數，表示目前配置不可行。
5. 若還有梯子，從最大堆取出目前最大的高度差，將那次攀升改用梯子，並把原先扣除的磚塊加回來。
6. 如果取回磚塊後仍不足且還有梯子，就繼續替換；若梯子耗盡後磚塊仍為負數，便無法抵達目前建築。
7. 每次資源仍可行時更新最遠索引。

.NET 的 `PriorityQueue<TElement, TPriority>` 預設會讓最小 priority 先出隊。為了模擬最大堆，程式以 `-difference` 作為 priority，因此最大的高度差會有最小的負 priority，並最先被取出。

### 範例演示

使用官方範例 1：`heights = [4,2,7,6,9,14,12]`、`bricks = 5`、`ladders = 1`。

表格中的「最大堆」列出仍暫時由磚塊負擔的正高度差：

| 移動 | 高度差 | 動作 | 剩餘磚塊 | 剩餘梯子 | 最大堆 | 最遠索引 |
| --- | ---: | --- | ---: | ---: | --- | ---: |
| `0 → 1` | `-2` | 免費下降 | `5` | `1` | `[]` | `1` |
| `1 → 2` | `5` | 先支付 5 塊磚 | `0` | `1` | `[5]` | `2` |
| `2 → 3` | `-1` | 免費下降 | `0` | `1` | `[5]` | `3` |
| `3 → 4` | `3` | 先扣 3 塊後不足；用梯子替換堆中最大的 `5`，取回 5 塊 | `2` | `0` | `[3]` | `4` |
| `4 → 5` | `5` | 扣 5 塊後不足，且沒有梯子可替換 | `-3` | `0` | `[5,3]` | 停在 `4` |

最後無法抵達索引 `5`，所以回傳 `4`。

### 為什麼正確

在任何掃描到的位置，最大堆保存了曾經暫定使用磚塊的正高度差。當磚塊不足時，將梯子替換到其中最大的高度差，會取回最多磚塊；任何改為替換較小高度差的方案都不可能留下更多磚塊。因此每次替換後，已走過區間的資源配置都是至少和其他配置同樣有利的。若連這個最佳配置都無法支付目前攀升，其他配置也不可能抵達更遠位置。

### 複雜度

- 時間複雜度：`O(n log n)`。每個正高度差最多入堆、出堆各一次。
- 空間複雜度：`O(n)`。最壞情況下最大堆保存所有正高度差。

## 解法二：最小堆，替梯子保留大落差

對應方法：`FurthestBuilding2(int[] heights, int bricks, int ladders)`

### 設計說明

這個解法不先扣除所有磚塊，而是直接維護「截至目前應該由梯子負擔的高度差」：

1. 從左到右計算相鄰建築的高度差，忽略非正高度差。
2. 每次遇到正高度差，先將它加入最小堆，暫定由梯子負擔。
3. 如果堆中元素數量沒有超過 `ladders`，代表每次攀升都還能分配一把梯子，不需要磚塊。
4. 如果堆中元素數量超過 `ladders`，代表至少有一次攀升必須改用磚塊。
5. 從最小堆取出目前最小的高度差並扣除相同數量的磚塊，使留在堆中的較大高度差繼續使用梯子。
6. 若磚塊變成負數，目前攀升無法完成，立刻回傳 `i - 1`。
7. 若完整掃描陣列，回傳最後一棟建築的索引。

當 `ladders = 0` 時，每個正高度差加入堆後都會立刻出隊並改用磚塊，同一套流程不需要額外分支。

### 範例演示

同樣使用官方範例 1。表格中的「最小堆」只保存暫定由梯子負擔的高度差：

| 移動 | 高度差 | 動作 | 剩餘磚塊 | 最小堆 | 可到達結果 |
| --- | ---: | --- | ---: | --- | --- |
| `0 → 1` | `-2` | 免費下降 | `5` | `[]` | 可到達 `1` |
| `1 → 2` | `5` | 將 `5` 暫定給唯一梯子 | `5` | `[5]` | 可到達 `2` |
| `2 → 3` | `-1` | 免費下降 | `5` | `[5]` | 可到達 `3` |
| `3 → 4` | `3` | 加入 `3` 後超出一把梯子；取出較小的 `3` 改付磚塊 | `2` | `[5]` | 可到達 `4` |
| `4 → 5` | `5` | 加入 `5` 後超出名額；取出一個 `5` 改付磚塊 | `-3` | `[5]` | 無法到達 `5` |

磚塊在 `4 → 5` 變成負數，因此回傳上一棟建築索引 `4`。

### 為什麼正確

掃描到任一位置時，最小堆最多保留 `ladders` 個高度差。每當數量超過限制，演算法都移除其中最小值並用磚塊支付，所以堆中留下的必然是截至目前最大的幾次攀升。這正是磚塊總成本最小的配置。若這個配置所需的磚塊仍超過可用數量，就不存在其他能抵達目前建築的資源分配方式。

### 複雜度

- 時間複雜度：`O(n log(ladders + 1))`；因為堆的大小最多短暫達到 `ladders + 1`。最壞情況 `ladders` 與 `n` 同階，因此也可寫成 `O(n log n)`。
- 空間複雜度：`O(ladders + 1)`，最壞為 `O(n)`。

## 兩種解法比較

| 比較項目 | 最大堆：`FurthestBuilding` | 最小堆：`FurthestBuilding2` |
| --- | --- | --- |
| 起始假設 | 每次攀升先使用磚塊 | 每次攀升先暫定使用梯子 |
| 調整時機 | 磚塊不足時 | 梯子名額超過時 |
| 堆保存內容 | 仍由磚塊負擔的正高度差 | 應由梯子負擔的最大高度差 |
| 出隊元素 | 最大高度差，改成梯子 | 最小高度差，改成磚塊 |
| 時間複雜度 | `O(n log n)` | `O(n log(ladders + 1))` |
| 額外空間 | `O(n)` | `O(ladders + 1)` |
| 優點 | 「缺磚就回收最大支出」容易從資源不足理解 | 堆較小，直接維持最佳梯子配置 |

兩種方法都不會修改 `heights`，也都只掃描陣列一次。若重視堆大小與參數化複雜度，最小堆版本較有優勢；最大堆版本則清楚展示如何對已做出的資源配置進行貪心替換。
