## 題目說明

給定整數陣列 `nums` 與整數 `k`，回傳排序後的第 `k` 大元素。

這裡的「第 k 大」不是第 `k` 個**相異**元素：重複值必須各自算一個排名。例如 `[5, 5, 4]` 的第 2 大仍是 `5`。

題目另外追問：能否不把整個陣列都排序？這正是 Quickselect 的切入點。

## 限制條件與重要定義

題目保證：

- `1 <= k <= nums.Length <= 10^5`
- `-10^4 <= nums[i] <= 10^4`

因此 `FindKthLargest`、`FindKthLargest2` 與 `FindKthLargest3` 假設輸入陣列非空，且 `k` 落在合法範圍內；它們不額外處理不合法輸入。

> [!WARNING]
> 三個解法都會**原地重排** `nums`：解法一會排序，解法二與解法三會交換分割區間中的元素。主控台範例在呼叫前會複製陣列，讓每個解法都從相同輸入開始。

## 解題出發點：先理解完整排序

若把陣列由小到大排好，第 `k` 大元素的零起始索引就是 `nums.Length - k`。例如 `nums = [3, 2, 1, 5, 6, 4]`、`k = 2`：

```text
升冪排序後： [1, 2, 3, 4, 5, 6]
陣列長度：   6
第 2 大索引： 6 - 2 = 4
答案：       nums[4] = 5
```

### 插入排序前情提要

插入排序把陣列左側視為「已排序前綴」。每次讀取下一個值（key），便由右往左移動比 key 大的元素，最後把 key 放入空出的正確位置。

以 `[3, 1, 2]` 為例：

| 處理的 key | 已排序前綴的變化 | 說明 |
| --- | --- | --- |
| `3` | `[3]` | 第一個元素天然有序。 |
| `1` | `[3]` → `[1, 3]` | `3` 右移，騰出位置給 `1`。 |
| `2` | `[1, 3]` → `[1, 2, 3]` | `3` 右移，`2` 插入 `1` 與 `3` 之間。 |

插入排序最壞與平均時間複雜度為 `O(n²)`，額外空間為 `O(1)`。本專案**沒有實作插入排序**；它在這裡是理解「先完整排序再取索引」的前情提要。實務上，解法一交由高效的 `Array.Sort` 處理；解法二與解法三則更進一步，只處理答案所在的半邊區間，差異在於 pivot 的選擇策略。

## 解法一：完整排序

對應方法：`FindKthLargest(int[] nums, int k)`

### 設計說明

1. 以 `Array.Sort(nums)` 將陣列原地升冪排序。
2. 由「第 `k` 大 = 升冪排序後索引 `nums.Length - k`」直接取值。
3. 不需要降冪排序，也不需要逐一走訪尋找第 `k` 個位置。

### 範例演示流程

以 `nums = [3, 2, 1, 5, 6, 4]`、`k = 2` 為例：

| 步驟 | 陣列或索引 | 結果 |
| --- | --- | --- |
| 升冪排序 | `[1, 2, 3, 4, 5, 6]` | 全部元素已定位。 |
| 換算目標索引 | `6 - 2 = 4` | 索引從 0 開始。 |
| 取值 | `nums[4]` | 回傳 `5`。 |

### 複雜度與取捨

| 時間 | 額外空間 | 優點 | 代價 |
| --- | --- | --- | --- |
| `O(n log n)` | 排序器遞迴堆疊約 `O(log n)`，陣列本身原地排序 | 實作短、容易閱讀、結果直觀 | 為了單一排名仍完成整體排序 |

當資料量不大、可讀性優先，或後續確實需要整個排序結果時，這是很可靠的選擇。

## 解法二：Quickselect

對應方法：`FindKthLargest2(int[] nums, int k)` 與私有輔助方法 `Quickselect`。

### 設計說明

Quickselect 沿用快速排序的「分割」概念，但不會遞迴排序左右兩側。它只保留目標索引所在的區間。

1. 先把第 `k` 大換成升冪排序後的 `targetIndex = nums.Length - k`。
2. 每輪固定以目前區間最左元素作為 `pivot`。
3. 用 Hoare 分割法讓左側元素不大於右側元素，並得到分割邊界 `j`。
4. 若 `targetIndex <= j`，只遞迴左半部；否則只遞迴右半部。
5. 區間縮小到單一元素時，該元素即為答案。

> [!IMPORTANT]
> Hoare 分割中的 `j` 是左右區間的**邊界**，不是 pivot 的最終索引。演算法正確性來自「目標索引只會落在其中一側」，而不是每一輪都固定 pivot 的最終排名。

### 範例演示流程

同樣使用 `nums = [3, 2, 1, 5, 6, 4]`、`k = 2`：

```text
targetIndex = 6 - 2 = 4
```

下表是目前「固定左端 pivot + Hoare 分割」實作的實際縮小過程：

| 目前區間 | pivot | 分割後陣列 | 邊界 `j` | 保留的下一區間 |
| --- | ---: | --- | ---: | --- |
| `0..5` | `3` | `[1, 2, 3, 5, 6, 4]` | `1` | `2..5`，因為 `4 > 1` |
| `2..5` | `3` | `[1, 2, 3, 5, 6, 4]` | `2` | `3..5`，因為 `4 > 2` |
| `3..5` | `5` | `[1, 2, 3, 4, 6, 5]` | `3` | `4..5`，因為 `4 > 3` |
| `4..5` | `6` | `[1, 2, 3, 4, 5, 6]` | `4` | `4..4`，因為 `4 <= 4` |
| `4..4` | — | — | — | 回傳 `nums[4] = 5` |

注意 Quickselect 不要求整個陣列在途中完全排序；上例最後碰巧有序，不是演算法的必要條件。

### 複雜度與取捨

| 時間 | 額外空間 | 優點 | 代價 |
| --- | --- | --- | --- |
| 隨機排列輸入平均 `O(n)`；最壞 `O(n²)` | 遞迴深度平均 `O(log n)`；最壞 `O(n)` | 不需完成整體排序，通常比完整排序少做工作 | 固定左端 pivot 遇到不利排列可能連續產生不平衡分割 |

這個版本刻意維持固定左端 pivot，讓分割流程可重現且容易追蹤。解法三會保留相同的 Hoare 分割，但改以隨機 pivot 降低連續遇到不平衡分割的機率。

## 解法三：隨機 Pivot Quickselect

對應方法：`FindKthLargest3(int[] nums, int k)` 與私有輔助方法 `QuickselectWithRandomPivot`。

### 設計說明

解法三刻意不改寫解法二，而是只替換 pivot 的選擇方式，方便直接比較兩者的取捨。

1. 同樣先換算 `targetIndex = nums.Length - k`，目標仍是升冪排序後的索引。
2. 每輪以 `Random.Shared.Next(left, right + 1)` 從目前候選區間隨機挑一個 pivot 索引。
3. 將該 pivot 與區間左端交換，讓後續 Hoare 分割可沿用解法二的掃描與邊界判斷。
4. 分割後依 `j` 判斷目標索引位於左側或右側，只遞迴該半邊區間。
5. 區間只剩一個元素時，該元素就是答案。

> [!IMPORTANT]
> 隨機 pivot 不改變答案，也不保證每一次執行都比固定 pivot 更快；它的作用是降低規律或刻意構造輸入反覆產生極端不平衡分割的機率。`Random.Shared` 沒有固定種子，因此中間分割路徑可能每次不同。

### 可能的範例演示流程

同樣使用 `nums = [3, 2, 1, 5, 6, 4]`、`k = 2`，因此 `targetIndex = 6 - 2 = 4`。以下**假設**隨機數依序選到索引 `4`、`1`、`3`；這只是一條可能的路徑，不是每次執行都會得到的追蹤結果。

| 目前區間 | 隨機選到的 pivot | 分割後陣列 | 邊界 `j` | 保留的下一區間 |
| --- | --- | --- | ---: | --- |
| `0..5` | 索引 `4`，值 `6` | `[4, 2, 1, 5, 3, 6]` | `4` | `0..4`，因為 `4 <= 4` |
| `0..4` | 索引 `1`，值 `2` | `[1, 4, 2, 5, 3, 6]` | `0` | `1..4`，因為 `4 > 0` |
| `1..4` | 索引 `3`，值 `5` | `[1, 3, 2, 4, 5, 6]` | `3` | `4..4`，因為 `4 > 3` |
| `4..4` | — | — | — | 回傳 `nums[4] = 5` |

每一輪都先把隨機選到的 pivot 換到區間左端，因此後續分割的程式結構與解法二相同；不同的是，解法三不會固定讓最左元素決定本輪分割品質。

### 複雜度與取捨

| 時間 | 額外空間 | 優點 | 代價 |
| --- | --- | --- | --- |
| 期望 `O(n)`；最壞 `O(n²)` | 遞迴深度期望 `O(log n)`；最壞 `O(n)` | 降低固定 pivot 對有序或規律輸入產生不平衡分割的風險 | 分割過程不可重現，最壞情況仍無法完全排除 |

相較於解法二，隨機 pivot 的優勢是期望效能更不容易受輸入排列影響；若需要完全可重現的教學或除錯路徑，固定左端 pivot 則更直觀。

## 實作注意事項

- `k` 是「由大到小」的排名，但三個實作都以升冪索引 `nums.Length - k` 思考，能避免額外反轉陣列。
- 重複元素不會被去重；排序與分割都把每個位置當成獨立排名。
- `RunSample` 對每種解法使用 `(int[])nums.Clone()`，因此主控台印出的輸入永遠保留原貌，也不會讓解法二或解法三的分割結果影響其他解法。
- 解法三使用 `Random.Shared` 選取 pivot；結果相同，但中間分割路徑可能不同。
- 這是單一 .NET 主控台專案，沒有 `dotnet test` 流程；請使用上方明示的專案路徑進行建置與範例驗證。
