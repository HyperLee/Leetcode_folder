# LeetCode 169 - Majority Element

以 C# 實作 LeetCode 169 `Majority Element`，並保留三種常見解法做對照：

- `Dictionary` 頻率統計
- `Sorting` 排序取中位數
- `Boyer-Moore Voting` 投票法

目前主程式會直接執行四組測試資料，輸出每種解法的結果，方便對照程式邏輯與文件說明。

## 題目連結

- LeetCode: <https://leetcode.com/problems/majority-element/description/>
- LeetCode 中文站: <https://leetcode.cn/problems/majority-element/description/>

## 題目說明

給定一個整數陣列 `nums`，請回傳其中的多數元素。

所謂多數元素，是指出現次數嚴格大於 `floor(n / 2)` 的元素，也就是它的出現次數超過整個陣列長度的一半。

題目額外保證：

- 陣列一定非空
- 多數元素一定存在

這個保證很重要，因為它會直接影響解法設計：

- `Dictionary` 不需要考慮「完全找不到答案」的情況
- `Sorting` 可以放心地取排序後的中間值
- `Boyer-Moore` 可以在最後直接回傳候選人，而不需要第二次驗證

## 限制條件

根據題目原始限制：

- `1 <= nums.length <= 5 * 10^4`
- `-10^9 <= nums[i] <= 10^9`
- 多數元素一定存在於陣列中

這些限制代表：

- 陣列長度最多到五萬，`O(n)` 與 `O(n log n)` 都可接受
- 由於元素值域很大，不能用「值當索引」的陣列計數法
- 題目保證多數元素存在，讓 Boyer-Moore 成為非常適合的最佳解

## 解題概念與出發點

這題的核心不是「找出最常出現的元素」而已，而是利用一個更強的條件：

> 某個值的出現次數，已經超過整個陣列的一半。

這個條件比「最大頻率」還強，因此可以推出幾個有用的性質：

1. 如果用 `Dictionary` 計數，只要某個值超過 `n / 2`，就可以直接回傳。
2. 如果先排序，多數元素一定會覆蓋排序後的中間位置。
3. 如果把不同元素互相抵消，多數元素因為數量最多，最後一定會留下來。

因此這份專案保留三種層次不同的思路：

1. `Dictionary`：最直觀、最好理解。
2. `Sorting`：開始利用數學性質與排序後的位置特徵。
3. `Boyer-Moore Voting`：不用額外空間，且時間最佳，是本題主解。

## 解法總覽

| 解法 | 核心想法 | 時間複雜度 | 空間複雜度 | 說明 |
| --- | --- | --- | --- | --- |
| Dictionary | 統計每個值出現次數 | `O(n)` | `O(n)` | 最直觀，但需要額外雜湊表 |
| Sorting | 排序後直接取中間值 | `O(n log n)` | 依排序實作而定，通常可視為 `O(log n)` | 利用多數元素覆蓋中位位置 |
| Boyer-Moore Voting | 不同元素互相抵消 | `O(n)` | `O(1)` | 本題最佳解 |

## 解法一：Dictionary 頻率統計

### 設計說明

這個方法最符合直覺：

1. 逐一走訪陣列中的每個數字
2. 用 `Dictionary<int, int>` 記錄這個數字目前出現了幾次
3. 每次更新次數後，立即檢查它是否已經超過 `floor(n / 2)`
4. 一旦超過，就可以立刻回傳

這種寫法的優點是很好理解，因為它直接把題目翻譯成程式：

- 題目問「誰出現超過一半」
- 我們就真的去數「每個人出現幾次」

因為題目保證一定存在多數元素，所以不需要在最後再花一輪找最大值，也不必處理無解情況。對於這題來說，只要某個值的計數第一次超過門檻，它就已經是答案。

### 範例演示流程

以 `nums = [2, 2, 1, 1, 1, 2, 2]` 為例：

- 陣列長度 `n = 7`
- 門檻 `floor(n / 2) = 3`
- 也就是說，誰的出現次數大於 `3`，誰就是答案

走訪過程如下：

1. 看到第一個 `2`
   `counts = { 2: 1 }`
   尚未超過 `3`
2. 看到第二個 `2`
   `counts = { 2: 2 }`
   尚未超過 `3`
3. 看到第一個 `1`
   `counts = { 2: 2, 1: 1 }`
   尚未超過 `3`
4. 看到第二個 `1`
   `counts = { 2: 2, 1: 2 }`
   尚未超過 `3`
5. 看到第三個 `1`
   `counts = { 2: 2, 1: 3 }`
   仍然沒有「大於」`3`
6. 看到第三個 `2`
   `counts = { 2: 3, 1: 3 }`
   仍然沒有「大於」`3`
7. 看到第四個 `2`
   `counts = { 2: 4, 1: 3 }`
   `4 > 3`，因此答案是 `2`

### 這個方法的重點

- 優點：簡單、穩定、容易第一次就寫對
- 缺點：要多開一個 `Dictionary`，空間不是常數
- 適合：建立第一版解題直覺，或在教學情境中先說明題意

## 解法二：Sorting 排序取中位數

### 設計說明

這個方法的關鍵不在「排序本身」，而在排序後產生的數學性質。

如果某個元素出現次數超過陣列長度的一半，那麼它在排序後一定會形成一大段連續區間，而且這段區間一定會覆蓋中間位置。

換句話說：

- 排序前：多數元素分散在陣列各處
- 排序後：相同元素會靠在一起
- 因為它超過一半，所以中間那格一定落在它身上

因此做法非常簡單：

1. 先排序
2. 回傳 `nums[nums.Length / 2]`

這個方法漂亮的地方在於，它不需要真的去數頻率，也不需要額外資料結構，而是直接利用多數元素「覆蓋中位索引」的性質求解。

### 為什麼中間值一定正確？

假設陣列長度是 `7`，中間索引是 `3`。

如果某個值出現超過一半，也就是至少出現 `4` 次，那它在排序後不可能完全待在左半邊，也不可能完全待在右半邊；它必然會跨過中間位置。

所以排序之後直接拿中間元素，就是答案。

### 範例演示流程

以 `nums = [2, 2, 1, 1, 1, 2, 2]` 為例：

1. 原始陣列：
   `[2, 2, 1, 1, 1, 2, 2]`
2. 排序後：
   `[1, 1, 1, 2, 2, 2, 2]`
3. 中間索引：
   `7 / 2 = 3`
4. `nums[3] = 2`
5. 因此答案是 `2`

再看一個更短的例子 `nums = [3, 2, 3]`：

1. 排序後變成 `[2, 3, 3]`
2. 中間索引 `3 / 2 = 1`
3. `nums[1] = 3`
4. 因此答案是 `3`

### 這個方法的重點

- 優點：思路簡潔，程式碼極短
- 缺點：需要排序，時間比 `O(n)` 解法慢
- 注意：排序會改動原陣列，所以主程式示範時要先複製陣列再排序

## 解法三：Boyer-Moore Voting Algorithm

### 設計說明

這是本題最經典、也最值得理解的解法。

它的核心想法不是「計數所有元素」，而是：

> 讓不同元素彼此抵消，最後剩下來的候選人，就是多數元素。

做法如下：

1. 維護一個 `candidate`，代表目前的候選多數元素
2. 維護一個 `count`，代表目前候選人的票數
3. 走訪陣列時：
   - 如果 `count == 0`，就把當前元素設成新的 `candidate`
   - 如果目前元素等於 `candidate`，`count++`
   - 如果不同於 `candidate`，`count--`

為什麼可行？

因為多數元素的數量比其他所有元素加起來都還多。即使它一路上和其他值互相抵消，最後仍然會有剩餘，因此最後留在 `candidate` 的值一定是它。

這也是為什麼 Boyer-Moore 能做到：

- 只走訪一次陣列
- 不需要 `Dictionary`
- 只用常數額外空間

### 範例演示流程

以 `nums = [2, 2, 1, 1, 1, 2, 2]` 為例：

初始狀態：

- `candidate = 0`
- `count = 0`

逐步走訪：

1. 看到 `2`
   `count == 0`，所以設 `candidate = 2`
   因為 `num == candidate`，`count = 1`
2. 看到 `2`
   與候選人相同，`count = 2`
3. 看到 `1`
   與候選人不同，`count = 1`
4. 看到 `1`
   與候選人不同，`count = 0`
   代表目前這一段的互相抵消已經打平
5. 看到 `1`
   `count == 0`，設 `candidate = 1`
   與候選人相同，`count = 1`
6. 看到 `2`
   與候選人不同，`count = 0`
7. 看到 `2`
   `count == 0`，設 `candidate = 2`
   與候選人相同，`count = 1`

最後留下的 `candidate = 2`，因此答案是 `2`

### 直觀理解

你可以把它想成「擂台賽」：

- 同陣營的人出現，幫擂主加一票
- 不同陣營的人出現，和擂主互相抵消一票
- 一旦票數歸零，就表示這一段勢均力敵，換新的候選人上台

只要題目保證真的存在多數元素，最後能站到最後的人就一定是它。

### 這個方法的重點

- 優點：`O(n)` 時間、`O(1)` 空間，是本題最佳解
- 缺點：第一次接觸時概念較抽象
- 適合：正式提交、面試最佳解、需要常數空間的情境

## 結論

這題最值得記住的不是「有三種寫法」，而是三種寫法分別代表不同層次的思考：

- `Dictionary`：直接把題意翻成計數
- `Sorting`：利用排序後的中位性質
- `Boyer-Moore Voting`：利用多數元素能在抵消後存活的本質

如果目標是第一次解出題目，`Dictionary` 很好上手；如果目標是理解題目的數學性質，`Sorting` 很有代表性；如果目標是正式提交最佳解，`Boyer-Moore Voting` 會是最值得優先掌握的方法。
