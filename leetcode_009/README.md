# LeetCode 9. Palindrome Number

以 C# 與 .NET 10 實作 LeetCode 第 9 題 Palindrome Number，並整理三種常見解法：

- 字串反轉
- 反轉一半數字
- 字串雙指針

這個專案的重點不是只得到答案，而是用同一題對照不同思考角度，理解每種寫法的出發點、判斷流程與取捨。

> [!NOTE]
> 專案目前在 [leetcode_009/Program.cs](leetcode_009/Program.cs) 中保留三種解法，並在 Main 中加入多組測試資料，方便直接執行觀察結果。

## Overview

題目要求判斷一個整數是否為回文數。

回文數的特性很單純：

- 從左到右讀，和從右到左讀完全相同。
- 例如 `121` 是回文，`123` 不是回文。
- 負數一定不是回文，因為負號只會出現在最左邊。

題目連結：

- LeetCode: [Palindrome Number](https://leetcode.com/problems/palindrome-number/description/)
- LeetCode 中文站: [回文數](https://leetcode.cn/problems/palindrome-number/description/)

## 題目核心觀察

在開始寫程式碼之前，先抓住幾個一定成立的事實：

1. 負數一定不是回文。
2. 若一個整數最後一位是 `0`，那它第一位也必須是 `0` 才可能是回文。
3. 但整數不會保留前導 `0`，所以只有 `0` 自己符合這件事。
4. 回文的本質是「對稱」，所以不一定要真的把整個數字完整反轉，能比較對稱位置也可以。

這四點，正是三種解法的共同出發點。

## 三種解法總覽

| 方法 | 核心概念 | 時間複雜度 | 空間複雜度 | 特點 |
| --- | --- | --- | --- | --- |
| `IsPalindrome` | 轉字串後完整反轉，再比對 | `O(n)` | `O(n)` | 最直觀，最容易理解 |
| `IsPalindrome2` | 只反轉一半數字 | `O(log_{10} n)` | `O(1)` | 最精煉，效率最佳 |
| `IsPalindrome3` | 轉字串後用左右指標比對 | `O(n)` | `O(n)` | 直觀且不用再建立完整反轉字串 |

> [!TIP]
> 如果是在面試或刷題場合，通常最值得優先掌握的是 `IsPalindrome2`，因為它直接利用數字特性，避免多餘轉換。

## 解法一：字串反轉

### 方法一的解題概念

先把整數轉成字串，再從尾巴一路往前組出反轉字串，最後和原字串比較。

如果兩者相同，代表正讀與反讀一致，就是回文。

### 方法一的出發點

這是最容易想到的寫法，因為「回文」這件事本來就很適合用字串視角理解。先把數字轉成字串，就能把問題變成字元序列的反轉與比對。

### 方法一的判斷流程

1. 將 `x` 轉成字串。
2. 從字串尾端往前走。
3. 逐步組出反轉後的字串。
4. 比較反轉字串與原字串是否相同。

### 方法一的優點與限制

- 優點是好懂、好寫、好驗證。
- 限制是需要額外空間保存反轉字串。

## 解法二：只反轉一半數字

### 方法二的解題概念

回文具有對稱性，因此不需要把整個數字都反轉，只要把後半段反轉後，和前半段比較即可。

例如 `1221`：

- 前半段是 `12`
- 後半段反轉後也是 `12`

例如 `12321`：

- 中間的 `3` 不影響回文判斷
- 只要比較 `12` 與 `12`

### 方法二的出發點

這個方法從「數字本身的位數結構」出發，而不是把問題先轉成字串。

每一次迴圈：

- 用 `% 10` 取出目前最低位數
- 接到 `revertedNumber` 後面
- 再把原數字用 `/= 10` 去掉最後一位

當原數字 `x` 小於或等於 `revertedNumber` 時，就代表已經處理到一半了。

### 方法二的判斷流程

1. 先排除負數。
2. 先排除結尾為 `0` 但本身不是 `0` 的數字。
3. 反轉後半段數字。
4. 若位數是偶數，比較 `x == revertedNumber`。
5. 若位數是奇數，比較 `x == revertedNumber / 10`。

### 方法二的優點與限制

- 優點是不用字串轉換，也不需要額外建立完整反轉結果。
- 優點是空間複雜度為 `O(1)`。
- 限制是相對抽象，第一次看不如字串法直覺。

## 解法三：字串雙指針

### 方法三的解題概念

先把整數轉成字串，再設定左右兩個指標：

- `left` 從最左邊開始
- `right` 從最右邊開始

每次比較兩端字元：

- 若不同，立即回傳 `false`
- 若相同，指標往中間縮進

只要一路都相同，最後就是回文。

### 方法三的出發點

這個方法保留了字串法的直觀性，但不需要另外建立一份完整反轉字串。它直接利用回文的對稱特性，逐對檢查對應位置。

### 方法三的判斷流程

1. 先排除負數。
2. 將數字轉成字串。
3. 以左右兩端開始比較。
4. 若有任一組不相同，立刻回傳 `false`。
5. 若一路比到中間都相同，回傳 `true`。

### 方法三的優點與限制

- 優點是邏輯清楚，而且可以提早結束。
- 優點是不用建立完整反轉字串。
- 限制是仍然需要把數字轉成字串。

## 三種方法流程演示

下面用具體範例，分別展示三種方法如何一步步得到結果。

### 方法一示範：`121`

原始字串：`121`

| 步驟 | 目前讀取字元 | 反轉結果 |
| --- | --- | --- |
| 1 | `1` | `1` |
| 2 | `2` | `12` |
| 3 | `1` | `121` |

最後比較：

- 原始字串：`121`
- 反轉字串：`121`
- 結果：`true`

### 方法二示範：`1221`

初始值：

- `x = 1221`
- `revertedNumber = 0`

| 迴圈次數 | `x % 10` | 更新後 `revertedNumber` | 更新後 `x` |
| --- | --- | --- | --- |
| 1 | `1` | `1` | `122` |
| 2 | `2` | `12` | `12` |

此時：

- `x = 12`
- `revertedNumber = 12`

因為 `x == revertedNumber`，所以結果為 `true`。

### 方法三示範：`12321`

字串內容：`12321`

| 步驟 | `left` 位置 | `right` 位置 | 比較結果 |
| --- | --- | --- | --- |
| 1 | `1` | `1` | 相同，繼續 |
| 2 | `2` | `2` | 相同，繼續 |
| 3 | 中間 `3` | 中間 `3` | 已完成比對 |

整個過程沒有發現不相同的對稱位置，因此結果為 `true`。

## 邊界情況

以下是這題最常見、也最容易漏掉的測試資料：

| 輸入 | 是否回文 | 原因 |
| --- | --- | --- |
| `121` | `true` | 正反都相同 |
| `-121` | `false` | 負號只存在左邊 |
| `10` | `false` | 反過來會像 `01`，但整數不保留前導 `0` |
| `0` | `true` | 單一數字本身就是回文 |
| `12321` | `true` | 中心對稱 |
| `123` | `false` | 左右不對稱 |

## 學習重點

如果你是第一次做這題，建議依照這個順序理解：

1. 先看字串反轉法，快速建立「什麼叫回文」的直覺。
2. 再看雙指針法，理解回文的核心其實是對稱比較。
3. 最後看只反轉一半數字，掌握不依賴字串的最佳化做法。

這樣不只會解這一題，也能把思路延伸到更多「對稱判斷」類型的題目。
