# LeetCode 148 - Sort List

這個專案整理 LeetCode 148「Sort List / 排序鏈結串列」的 C# console app 解法。程式內含可直接執行的 linked-list 範例資料，並以自底向上的迭代歸併排序完成 `O(n log n)` 排序，同時維持常數額外空間的指標操作思路。

題目連結：

- [LeetCode - Sort List](https://leetcode.com/problems/sort-list/description/)
- [LeetCode CN - 排序鏈結串列](https://leetcode.cn/problems/sort-list/description/)

## 題目說明

給定一個 singly linked list 的頭節點 `head`，請將整條 linked list 依照遞增順序排序後回傳新的頭節點。

題目的 follow-up 也明確要求：

- 時間複雜度要做到 `O(n log n)`。
- 額外空間要盡量維持 `O(1)`。

這使得直接把資料複製到陣列後排序雖然可行，但不符合這題想考的 linked-list 指標操作核心；真正貼近題意的做法是 merge sort。

## 限制條件

- `0 <= 節點數量 <= 5 * 10^4`
- `-10^5 <= Node.val <= 10^5`

## 解題概念與出發點

這題的關鍵不只是「排序」，而是「在 linked list 上排序」。

如果資料結構是陣列，我們可以隨時用索引跳到任意位置；但 linked list 只能沿著 `next` 一步一步往後走，因此像 quicksort 這種常依賴隨機存取或大量 swap 的寫法不夠自然。相對地，merge sort 天生只需要：

- 把串列切成數段
- 比較兩段目前的頭節點
- 重新接回 `next` 指標

這些操作都非常適合 singly linked list。

本專案採用的不是遞迴版 merge sort，而是**自底向上（bottom-up）的迭代歸併排序**。原因是：

1. 題目追求 `O(1)` 額外空間。
2. 遞迴版 merge sort 雖然時間複雜度也正確，但會額外消耗呼叫堆疊。
3. 迭代版只透過幾個指標變數控制切割與合併，更貼近 follow-up 想考的常數空間思路。

整體想法可以想成：

- 先把每個單一節點視為長度 `1` 的已排序區段。
- 第一輪把每兩段長度 `1` 的區段合併成長度 `2` 的已排序區段。
- 第二輪把每兩段長度 `2` 的區段合併成長度 `4` 的已排序區段。
- 持續把區段長度倍增成 `8`、`16`、`32`...
- 當區段長度大於等於整體鏈結串列長度時，整條串列就已排序完成。

## 解法設計說明

方法：`SortList`

### 核心設計

這個解法由三個關鍵動作組成：

1. **先計算整條 linked list 長度**
   
   需要先知道總長度，才能控制外層迴圈中的 `step` 何時停止倍增。

2. **用 `SplitList` 切出固定長度區段**
   
   每一輪都要從目前位置往後切出兩段長度最多為 `step` 的子串列，準備交給合併流程。

3. **用 `MergeTwoLists` 合併兩段已排序區段**
   
   合併完成後不只要拿到新頭節點，也要拿到新尾節點，這樣才能在 `O(1)` 時間內把下一段結果接上，不必每次都重新掃到尾端。

### 為什麼要回傳 merged tail

這是迭代版寫法最重要的細節之一。

如果每次合併完只知道 merged head，不知道 merged tail，外層就得從 merged head 再走一次，找到尾端後才能把下一組合併結果接上。那麼每一輪都會額外做很多重複掃描，程式會變得更慢也更混亂。

因此 `MergeTwoLists` 直接回傳：

- 合併後串列的頭節點
- 合併後串列的尾節點

如此一來，`newListTail` 可以永遠指向「目前新串列的最後一個節點」，讓整體串接過程維持線性推進。

### 為什麼 `SplitList` 很重要

自底向上 merge sort 的核心不是「知道怎麼 merge」，而是「每一輪都要正確切出兩段 step 長度的已排序區段」。

`SplitList(head, size)` 會做的事是：

- 從 `head` 開始往後走最多 `size` 個節點。
- 將這一段與後面的節點斷開。
- 回傳後半剩餘串列的頭節點。

這個設計讓外層排序流程可以非常規律地做：

1. 取出第一段 `head1`
2. 切出第二段 `head2`
3. 再切出下一輪的起點 `current`
4. 合併 `head1` 與 `head2`

### 複雜度分析

- 時間複雜度：`O(n log n)`
  
  每一輪會走訪整條串列一次，而 `step` 會從 `1` 倍增到大於等於 `n`，總共約 `log n` 輪。

- 額外空間：`O(1)`
  
  只使用少量指標變數與哨兵節點，沒有額外建立與 `n` 成比例的資料結構。

## 範例演示流程

### 範例一：`[4, 2, 1, 3]`

初始串列：

```text
4 -> 2 -> 1 -> 3
```

#### 第 1 輪：`step = 1`

此時每個單一節點都視為已排序區段：

- 第一組：`[4]` 與 `[2]`
- 第二組：`[1]` 與 `[3]`

分別合併後得到：

- `[4] + [2] -> [2, 4]`
- `[1] + [3] -> [1, 3]`

整條串列變成：

```text
2 -> 4 -> 1 -> 3
```

#### 第 2 輪：`step = 2`

此時每兩個節點為一組已排序區段：

- 第一組：`[2, 4]`
- 第二組：`[1, 3]`

合併流程：

1. 比較 `2` 和 `1`，先接 `1`
2. 比較 `2` 和 `3`，接 `2`
3. 比較 `4` 和 `3`，接 `3`
4. 第二段用完，接回第一段剩餘的 `4`

結果：

```text
1 -> 2 -> 3 -> 4
```

因為下一輪 `step = 4` 已不小於串列長度 `4`，排序完成。

### 範例二：`[-1, 5, 3, 4, 0]`

這組範例的重點是觀察**尾端不足完整 step 長度時如何處理**。

初始串列：

```text
-1 -> 5 -> 3 -> 4 -> 0
```

#### 第 1 輪：`step = 1`

切出的區段會是：

- `[-1]` 與 `[5]`
- `[3]` 與 `[4]`
- `[0]` 與 `null`

合併後得到：

```text
-1 -> 5 -> 3 -> 4 -> 0
```

前兩組各自排序後其實沒有變化；最後 `[0]` 因為沒有第二段可合併，就保留到下一輪。

#### 第 2 輪：`step = 2`

切出的區段變成：

- `[-1, 5]` 與 `[3, 4]`
- `[0]` 與 `null`

第一組合併：

1. `-1` 比 `3` 小，先接 `-1`
2. `5` 比 `3` 大，接 `3`
3. `5` 比 `4` 大，接 `4`
4. 第二段用完，接回剩餘的 `5`

得到：

```text
-1 -> 3 -> 4 -> 5 -> 0
```

尾端的 `[0]` 依舊保留。

#### 第 3 輪：`step = 4`

此時切出的區段是：

- `[-1, 3, 4, 5]`
- `[0]`

合併時：

1. `-1` 比 `0` 小，先接 `-1`
2. `3` 比 `0` 大，接 `0`
3. 第二段用完，把第一段剩餘的 `3 -> 4 -> 5` 全部接回

最後結果：

```text
-1 -> 0 -> 3 -> 4 -> 5
```

這正是程式內建範例中的預期輸出。
