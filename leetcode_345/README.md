# LeetCode 345 - Reverse Vowels of a String

這個專案是 LeetCode 345「Reverse Vowels of a String」的 C# Console 範例。程式會在 `Main` 中直接執行固定測資，展示如何只反轉字串中的母音字母，同時保留所有非母音字元的原始位置。

## 題目說明

給定一個字串 `s`，請只反轉其中所有母音字母，並回傳處理後的新字串。

題目定義的母音只有：

- `a`
- `e`
- `i`
- `o`
- `u`
- 以及它們對應的大寫字母

重點不是把整個字串反轉，而是只改變母音之間的先後順序。所有非母音字元都必須留在原本的位置上。

範例：

```text
輸入：s = "IceCreAm"
輸出："AceCreIm"
```

原本的母音順序是 `I, e, e, A`，反轉後會變成 `A, e, e, I`，再回填到原本的母音位置。

## 限制條件

- `1 <= s.length <= 3 * 10^5`
- `s` 只會由可列印的 ASCII 字元組成

## 解題概念與出發點

這題最自然的出發點是先觀察：

- 我們只在意母音的位置與母音本身
- 非母音字元完全不需要搬動
- 如果每次都從頭重新尋找下一個母音，會造成不必要的重複掃描

因此最佳切入方式是使用兩個指標：

- 左指標從字串開頭往右走
- 右指標從字串結尾往左走

兩個指標都只在「找到母音」時停下來。當左右指標都停在母音上，就交換這兩個字元，然後繼續向中間收斂。

這個方法的優點是：

- 每個字元最多被左右指標看過一次
- 不需要另外儲存所有母音的位置
- 能直接在字元陣列上完成交換，邏輯清楚

## 解法一：雙指標

### 核心設計

實作流程如下：

1. 先把字串轉成 `char[]`，因為字串本身不可變，交換字元時需要可寫入的容器。
2. 設定兩個索引：
   - `left = 0`
   - `right = s.Length - 1`
3. 當 `left < right` 時重複執行：
   - 讓 `left` 一直往右走，直到它停在母音，或與 `right` 相遇
   - 讓 `right` 一直往左走，直到它停在母音，或與 `left` 相遇
   - 如果此時仍有 `left < right`，代表兩邊都找到了可以互換的母音
4. 交換兩個母音後，同步把 `left` 往右、`right` 往左，繼續處理下一組
5. 迴圈結束後，把字元陣列重新轉回字串

### 為什麼這個方法有效

因為母音的反轉，本質上就是：

- 最左邊的母音，要和最右邊的母音交換
- 次左邊的母音，要和次右邊的母音交換
- 一直做到所有母音都完成配對

這正好就是雙指標最擅長的問題型態。它不需要額外整理所有母音再回填，而是直接在掃描過程中完成對應交換。

### 判斷母音的方式

在這份實作裡，`IsVowel` 會檢查字元是否存在於字串 `"aeiouAEIOU"` 中。這樣可以同時支援：

- 小寫母音
- 大寫母音
- 混合大小寫的輸入

### 實作細節

- `ReverseVowels(string s)` 是對外的主要解法入口
- `IsVowel(char character)` 專門負責判斷是否為母音
- `Swap(char[] characters, int left, int right)` 專門負責交換兩個索引位置的字元
- `RunSamples()` 會執行固定案例，讓 `dotnet run` 直接成為這個 repo 的驗證方式

## 替代思路比較

這個 repo 只實作雙指標版本，但如果從設計角度比較，還有幾種常見想法：

### 思路 A：先收集所有母音，再反向回填

做法：

1. 先掃描整個字串，把所有母音依序收集到一個清單
2. 第二次掃描字串，遇到母音時就從清單尾端取值回填

優點：

- 寫法直觀
- 比較容易從「先找資料，再回寫」的角度理解

缺點：

- 需要額外 `O(k)` 空間儲存所有母音，其中 `k` 是母音數量
- 邏輯會分成兩次掃描與兩套狀態管理

### 思路 B：記錄所有母音索引，再成對交換

做法：

1. 先掃描一次，把所有母音位置記到索引陣列
2. 再用頭尾夾擊方式，依序交換這些索引位置上的字元

優點：

- 容易清楚看見「哪些位置是母音」
- 對教學來說，可視化母音位置會比較直接

缺點：

- 仍然需要額外儲存索引
- 和直接雙指標相比，多了一層資料結構管理

### 為什麼本專案選擇雙指標

因為題目只要求把母音反轉，沒有要求保留額外資訊。雙指標可以：

- 一次線性掃描完成
- 不必額外記錄全部母音
- 在空間與可讀性之間取得很好的平衡

對這題來說，雙指標是最貼近題目本質的解法。

## 範例演示流程

### 範例一：`"IceCreAm"`

原始字串：

```text
I c e C r e A m
0 1 2 3 4 5 6 7
```

步驟：

1. `left = 0`，停在 `I`，因為它是母音
2. `right = 7`，`m` 不是母音，往左移到 `A`
3. 交換 `I` 與 `A`

交換後：

```text
A c e C r e I m
```

4. `left` 往右、`right` 往左
5. 左側下一個母音是索引 `2` 的 `e`
6. 右側下一個母音是索引 `5` 的 `e`
7. 交換 `e` 與 `e`，字串外觀不變，但代表這組母音配對完成
8. 指標再收斂後相遇，流程結束

結果：

```text
"AceCreIm"
```

### 範例二：`"leetcode"`

母音依序是：

```text
e, e, o, e
```

反轉後變成：

```text
e, o, e, e
```

回填後得到：

```text
"leotcede"
```

雙指標觀點下的關鍵是：

- 第一組交換的是左邊第一個 `e` 與右邊最後一個 `e`
- 第二組交換的是左邊第二個 `e` 與右邊的 `o`

### 範例三：`"Aa"`

這個案例用來驗證大小寫混合時是否仍能正確辨識母音。

流程：

1. 左邊停在 `A`
2. 右邊停在 `a`
3. 交換後得到 `"aA"`

這說明大小寫母音都被納入同一套判斷規則。

### 範例四：`"bcdfg"`

這個字串沒有任何母音，因此：

- 左指標一路往右跳過所有字元
- 右指標不會找到可配對母音
- 沒有任何交換發生

最後結果與輸入完全相同：

```text
"bcdfg"
```

### 範例五：`"queueing"`

這個案例適合觀察「指標會跳過中間大量非目標位置」的行為。

母音位置與內容：

```text
q u e u e i n g
  ^ ^ ^ ^ ^
```

母音順序是：

```text
u, e, u, e, i
```

反轉後會變成：

```text
i, e, u, e, u
```

回填後得到：

```text
"qieueung"
```

這個例子顯示：

- 左右指標不需要關心每個子音
- 只要確保最後停在母音位置即可
- 母音數量是奇數時，中間那個母音會自然保留在正中央配對位置

## 複雜度分析

設字串長度為 `n`。

- 時間複雜度：`O(n)`
  - 左右指標都只會單向移動，每個字元最多被檢查有限次
- 空間複雜度：`O(n)`
  - 這份 C# 實作使用 `char[]` 保存可修改字元內容，因此需要與輸入等長的字元陣列

如果只從額外輔助結構的角度看，雙指標本身沒有再建立與母音數量成比例的額外集合。
